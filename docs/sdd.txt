1 Introduction 
1.1 Purpose 
The purpose of this Software Design Description (SDD) is to define the technical design of the P2P 
Dynamic Port Switching system, including its architecture, modules, interfaces, and 
implementation-level structure. This document is intended for developers and reviewers who need 
to understand how the system will be built in C using low-level socket operations and how protocol 
components interact for asymmetric bootstrap exchange, deterministic port computation, 
deterministic role assignment, connection handling, authentication, and logging. 
 
In this design, only asymmetric encryption is used for secure initialization. The inputs that start the 
port-generation process are transmitted to the remote peer through an asymmetric bootstrap 
exchange before the dynamic port switching cycle begins. No symmetric secret key or session key is 
generated, exchanged, or used by the system. 
 
1.2 Scope 
This SDD covers the design of a two-peer (Peer A and Peer B) communication system in which 
both peers dynamically switch ports using a deterministic time-step and a common bootstrap 
parameter set established through asymmetric encryption during initialization. The design 
includes: 
 
• Peer-to-peer architecture (no central application server required) 
• Transport support for TCP and UDP 
• Asymmetric bootstrap exchange for initialization parameters 
• Deterministic port, role, and protocol computation 
• Logging, configuration, and error handling 
 
This SDD does not cover: 
 
• GUI screens (this is a nice to have feature) 
• Large-scale multi-peer mesh networking 
• Any symmetric key distribution or symmetric session encryption mechanism

1.6 Operational Assumptions and Out-of-Band Configuration 
To preserve deterministic Dynamic Port Switching behavior while avoiding manual secret 
sharing, the system uses an asymmetric bootstrap model. Before the port-hopping cycle 
begins, the initiating peer transmits the Bootstrap Parameter Set to the remote peer using the 
remote peer’s RSA/ECC public key. The receiving peer decrypts the payload using its private key 
and sends an acknowledgement after successful validation. 
The system does not generate, exchange, or rely on any symmetric secret key or session key. All 
secure initialization is performed only through asymmetric encryption. 
The following assumptions apply: 
• 
IP Address Knowledge: Both peers must know each other’s network address (IPv4) 
before initialization. The current version of the system does not use a discovery server or 
STUN/TURN services for initial peer location. 
 
• 
Asymmetric Bootstrap Exchange: Before the dynamic port switching cycle begins, the 
initiating peer constructs a Bootstrap Parameter Set and encrypts it with the remote 
peer’s public key. The payload is then transmitted over the bootstrap channel and can 
only be decrypted by the intended peer holding the corresponding private key.

• 
Bootstrap Parameter Set Content: The bootstrap payload must include the bootstrap 
input count and all parameters required to start deterministic port generation. At 
minimum, this includes: 
- bootstrap_input_count 
- timing_interval 
- port_range_min 
- port_range_max 
- peer_id information 
- optional bootstrap_epoch or start time reference, if used by the implementation 
 
• 
Unique Peer ID Assignment: To avoid role collision, each peer must have a unique ID (0 
or 1). These identifiers are used by the deterministic role assignment logic to ensure that 
one peer listens while the other dials for each time-step. 
 
• 
Parameter Consistency: Dynamic port generation can only begin after both peers 
operate on the same validated Bootstrap Parameter Set. Any mismatch in bootstrap 
parameters will lead to synchronization failure because the calculated target ports and 
role outputs will no longer align.

2.2 Software & Environment Requirements 
• 
Operating System (OS): 
o Primary Support: Unix-based systems (Linux distributions such as Ubuntu, 
Debian, Fedora, Kali Linux) and macOS. 
o Note on Windows: Due to OS-level restrictions on Raw Socket creation 
(specifically regarding TCP headers), native Windows support is excluded. 
Windows users are advised to run the application within a Virtual Machine (VM) 
or WSL2 (Windows Subsystem for Linux). 
• 
Network Protocol Support: The operating system must natively support the TCP/IP 
Protocol Suite (IPv4). 
• 
Time Synchronization: Active NTP (Network Time Protocol) service is mandatory. The 
system clock must be synchronized to ensure the drift remains within the tolerated 
timing-interval. 
• 
User Privileges: The application requires Administrator (Windows) or Root/Sudo 
(Linux/macOS) privileges to access Raw Sockets and bind to specific ports 
dynamically. 
• 
Dependencies: 
o C Runtime Library (CRT) for the target OS. 
o System libraries for GUI rendering (e.g., GTK, Qt, or native Windows API 
depending on implementation). 
o OpenSSL / libcrypto: Required for generating cryptographic keys, performing 
RSA/ECC asymmetric encryption, and managing the secure initial handshake.

4 High Level Design 
At a high level, both peers run the same executable and follow the same deterministic algorithm 
to: 
• 
compute port and role, 
• 
establish a TCP/UDP connection, 
• 
perform mutual authentication, 
• 
exchange data, 
• 
log and continue. 
4.1 Module 1 — Communication Controller 
Brief Description: This module acts as the central orchestrator for the Transport Layer 
operations. It is responsible for maintaining the state of the active protocol (TCP or UDP) and 
managing the transition between them. It executes the actual switching logic based on 
commands from the Protocol Decision Logic module or user manual input. 
Responsibilities: 
• 
Protocol Lifecycle Management: Handles the initialization, binding, and teardown of 
TCP sockets and UDP endpoints. 
• 
State Retention: Stores the currently active protocol flag (e.g., ACTIVE_PROTO = TCP) to 
ensure the rest of the system utilizes the correct transport method for the current 
interval. 
• 
Asymmetric Bootstrap Orchestration: Manages the initial STATE_ASYM_BOOTSTRAP 
before the main communication loop. This includes selecting a bootstrap port, 
transmitting the Bootstrap Parameter Set using the remote peer’s public key, validating 
the acknowledgement, and releasing the system into the deterministic communication 
loop. No symmetric key is generated, exchanged, or stored. 
• 
Switch Execution: 
o Manual Switch: Instantly changes protocol upon user request via CLI/GUI. 
o Automatic Switch: Executes protocol transitions triggered by the adaptive logic, 
ensuring seamless handover (e.g., draining TCP buffers before switching to 
UDP). 
4.2 Module 2 — Protocol Decision Logic 
Brief Description: This module implements the intelligence behind the Adaptive Mode. It 
continuously analyzes network conditions (latency, packet loss) and uses the validated 
Bootstrap Parameter Set together with the current Time-Step to make deterministic protocol 
decisions for the next interval. 
Responsibilities:

Page 10 
 
• 
Network Analysis: Monitors round-trip time (RTT) and packet delivery success rates 
to classify the network state (for example, Stable, Congested, Critical). 
• 
Dynamic Weight Adjustment: Adjusts the selection probability ratio based on 
network classification. 
• 
Deterministic Decision Making: Uses the Bootstrap Parameter Set together with the 
current Time-Step and the calculated decision threshold to deterministically select 
the protocol. This ensures that both peers arrive at the same protocol decision 
without explicit negotiation, provided that they observe the same network state 
model. 
4.3 Module 3 — Port Management Component 
Brief Description: This module is responsible for the execution mechanism of the Dynamic Port 
Switching logic. Unlike the decision modules, this component handles the actual "heavy lifting" 
of binding to new ports and managing the transition lifecycle. 
Responsibilities: 
• 
Deterministic Port Computation: Calculates the active communication port for the 
current interval using the Bootstrap Parameter Set and the current Time-Step. It 
supports both cryptographic hash-based mapping and range-based port selection. 
• 
Re-Rendezvous Execution: Manages the critical "Handover Phase" where the system 
transitions from the old port to the new port. This involves tearing down the previous 
socket and initializing the new one. 
• 
Connection Verification: Enforces a "Verify-then-Transmit" policy. Even if a port is 
open, the system halts application data transmission until the mutual handshake on the 
new port is fully verified. 
 
4.4 Module 4 — Timing & Synchronization Handler 
Brief Description: 
This module acts as the "heartbeat" of the system. It abstracts the operating system's time 
functions and ensures that the application runs in lock-step with the configured time intervals. 
Responsibilities: 
• 
NTP Reliance: Relies on the host Operating System’s Network Time Protocol (NTP) 
service. It assumes that both peers are reasonably synchronized to a global clock 
source. 
• 
Time-Step Calculation: continuously calculates the current Time-Step index(T = 
CurrentTime / Interval). 
• 
Event Triggering: Serves as an interrupt generator. It monitors the clock and triggers 
signals for: 
o Port Hopping: When the Time-Step increments. 
o Protocol Re-evaluation: Periodically triggering the Protocol Decision Logic to 
check if network conditions have changed.

4.5 Module 5 — Role Assignment Component 
Brief Description: This module deterministically assigns the network role (Listener or Dialer) to 
each peer for every time-step. It ensures that for any given interval, one peer acts as the server 
while the other acts as the client without manual negotiation.  
Responsibilities:  
• 
 Deterministic Role Calculation: Uses the Bootstrap Parameter Set and the current 
Time-Step to calculate which peer ID should listen and which should dial. 
• 
Collision Avoidance: Guarantees that both peers do not attempt to take the same role 
simultaneously, preventing connection failures.  
 
4.6 Module 6 — Logging Component 
Brief Description: A specialized monitoring module designed to capture high-value operational 
metrics and state transitions without recording sensitive authentication secrets. It implements 
the Hybrid Logging mechanism (Console + File). 
Responsibilities: 
• 
State Transition Logging: Records every instance of a protocol switch (e.g., TCP -> UDP) 
and the rationale behind it. 
• 
Adaptive Logic Insights: Specifically logs "Skipped TCP Attempts" when the system 
operates in UDP mode due to high congestion (e.g., "[INFO] Decision: UDP selected 
(80% weight). TCP handshake skipped."). 
• 
Performance Metrics: Logs the latency of the re-connection process, specifically the 
SYN-ACK Time Interval (the time elapsed between sending a SYN packet to a new 
dynamic port and receiving the acknowledgment). 
• 
Port History: Maintains a sequential record of all computed and utilized ports for post-
experiment auditing.

5 Low Level Design 
5.1 Communication Controller (Flow) 
• 
1. Initialization: 
1. System Startup: Load local keys and initialize network loggers. 
2. Asymmetric Bootstrap Phase: 
▪ 
Dialer constructs the Bootstrap Parameter Set. 
▪ 
The Bootstrap Parameter Set includes the bootstrap input count and all 
fields required to start deterministic port generation. 
▪ 
Dialer encrypts the payload with the Listener’s public key and sends 
MSG_ASYM_BOOTSTRAP. 
▪ 
Listener decrypts the payload with its private key, validates all required 
fields, stores the Bootstrap Parameter Set as active configuration, and 
sends an ACK. 
▪ 
No symmetric secret key or session key is generated or exchanged. 
• 
2. Interval Loop (Main Thread): 
o Check Time: Is Current_Time_Step > Last_Time_Step? 
▪ 
YES (Switch Event): 
• 
Pause Outgoing Queue. 
• 
Call Protocol Decision Logic -> Get Target_Protocol (TCP/UDP). 
• 
Call Role Assignment -> Get Role (Listener/Dialer). 
• 
Call Port Management -> Get Target_Port. 
• 
Execute Switch -> Verify New Connection. 
• 
Flush Buffer: Send all queued messages to the new port. 
• 
Resume Outgoing Queue. 
• 
Handle I/O: 
o If Incoming_Packet -> Pass to UI (Chat History). 
o If User_Input -> Push to Outgoing Queue. 
5.2 Protocol Decision Logic (Flow) 
• 
Input: Packet_Loss_Rate, Avg_Latency 
• 
1. Determine Network State:

o If Latency > 200ms OR Loss > 5% -> Set State = CONGESTED. 
o Else -> Set State = STABLE. 
• 
2. Set Probability Weight: 
o If STABLE -> Set UDP_Threshold = 50 (50% chance). 
o If CONGESTED -> Set UDP_Threshold = 20 (80% chance for UDP).

5.3 Port Managament Component (Flow) 
• 
Input: Time-Step, Port_Range (Min, Max). 
• 
1. Computation: 
o Hash_Val = SHA256(Bootstrap_Parameter_Set + Time-Step). 
o Offset = ConvertToInt(Hash_Val) % (Max - Min). 
o Target_Port = Min + Offset. 
• 
2. Execution (Handover): 
• 
Teardown: shutdown(old_socket, RDWR) -> close(old_socket). 
• 
Re-Initialization: 
o Create new Raw Socket. 
o If Role is Listener -> bind(Target_Port). 
o If Role is Dialer -> connect(Target_IP, Target_Port). 
5.4 Timing & Synchronization Handler (Flow) 
• 
1. Sync Check: 
o Query System_Clock (NTP Synchronized). 
• 
2. Step Calculation: 
o Current_Step = Floor(System_Time / Configured_Interval). 
• 
3. Trigger Logic: 
o Compare Current_Step with Stored_Step. 
o If Different -> Return TRUE (Trigger Port Switch). 
o Else -> Return FALSE.

5.7 Packet Reliability Mechanisms (UDP Mode) 
To mitigate packet loss inherent in UDP and during port transition phases, the system 
implements the following strategies: 
• 
Packet Redundancy (Dual-Send): For critical chat messages in UDP mode, the system 
optionally dispatches the same datagram twice in rapid succession. This statistical 
approach reduces the probability of loss without requiring a complex handshake. 
• 
Application-Layer Acknowledgement: Upon receiving a DATA packet, the receiver 
immediately dispatches an ACK packet (Message Type 3) back to the sender. If the 
sender does not receive an ACK within a defined timeout (e.g., 500ms), the message is 
re-transmitted.  

SENG 491 – 492 Graduation Project SDD 
Page 15 
 
• 
 Transition Buffering: During the "Handover Phase" (port switching), the 
Communication Controller pauses the outgoing transmission queue. Messages 
generated during this micro-window are buffered in local memory and flushed to the 
new port once the re-handshake is verified. 
6 Bootstrap Parameter Set Definition 
The Bootstrap Parameter Set is the only initialization object used to start deterministic port 
generation. It is transmitted through asymmetric encryption before the first communication 
interval begins. 
 
This parameter set shall include the bootstrap input count and every value required by the port-
generation logic. No symmetric secret key, randomization secret, or session key is permitted in 
this design. 
 
After successful decryption and validation of the Bootstrap Parameter Set, both peers begin the 
deterministic time-step based port switching cycle using the same validated inputs.
